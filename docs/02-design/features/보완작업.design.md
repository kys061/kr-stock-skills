## 1. 설계 개요

3개 스킬의 기존 모듈에 컴포넌트/패턴을 추가하는 패치 작업. 하위 호환성 100% 유지가 핵심 제약.

## 2. Item 1: TGA 컴포넌트 — `liquidity_tracker.py`

### 2.1 현재 구조

```
LIQUIDITY_WEIGHTS = {fed_bs: 0.30, m2: 0.30, dxy: 0.25, rrp: 0.15}  # sum=1.00
track_liquidity(fed_bs_change_pct, m2_growth_yoy, dxy_change_3m, rrp_change_pct)
```

### 2.2 변경 설계

#### 새 상수

```python
# TGA 스코어링 (TGA 감소 = 유동성 증가)
TGA_SCORING = {
    'large_drawdown': 80,   # 1M change < -10%
    'drawdown': 65,         # 1M change -3% ~ -10%
    'stable': 50,           # 1M change -3% ~ +3%
    'buildup': 35,          # 1M change +3% ~ +10%
    'large_buildup': 20,    # 1M change > +10%
}
```

| 상수명 | 값 | 설명 |
|--------|-----|------|
| `TGA_SCORING['large_drawdown']` | 80 | TGA 1M < -10% → 유동성 확대 |
| `TGA_SCORING['drawdown']` | 65 | TGA 1M -3%~-10% |
| `TGA_SCORING['stable']` | 50 | TGA 1M +-3% |
| `TGA_SCORING['buildup']` | 35 | TGA 1M +3%~+10% |
| `TGA_SCORING['large_buildup']` | 20 | TGA 1M > +10% → 유동성 축소 |

#### 가중치 변경 (조건부)

| 입력 범위 | 반환값 | 해석 |
|----------|--------|------|
| < -10% | 80 | large_drawdown: 유동성 크게 확대 |

#### `track_liquidity()` 시그니처 변경

```python
def track_liquidity(fed_bs_change_pct=0.0, m2_growth_yoy=0.0,
                    dxy_change_3m=0.0, rrp_change_pct=0.0,
                    tga_change_1m_pct=None):    # NEW: default=None (하위 호환)
```

- `tga_change_1m_pct=None` → 기존 4-컴포넌트 가중치 사용 (LIQUIDITY_WEIGHTS)
- `tga_change_1m_pct=<float>` → 5-컴포넌트 가중치 사용 (LIQUIDITY_WEIGHTS_5)
- `components` dict에 `tga` 키 추가 (제공 시만)

## 3. Item 2: 고객 예탁금 — `credit_balance_analyzer.py`

### 3.1 현재 구조

이미 `DEPOSIT_CREDIT_RATIO` 상수와 `calc_deposit_credit_ratio()` 함수가 존재. 신용잔고 대비 예탁금 비율 분석은 구현 완료.

### 3.2 추가 설계: 예탁금 독립 분석 함수

기존 함수는 "예탁금 대비 신용잔고"만 분석. **예탁금 자체의 증감 트렌드** 분석이 누락.

#### 새 상수

```python
# 고객 예탁금 수준 (조원)
DEPOSIT_LEVEL_THRESHOLDS = {
    'very_high': 60.0,    # 60조원+: 대기 자금 풍부 (시장 바닥 근처)
    'high': 50.0,         # 50조원+
    'normal': 40.0,       # 40조원+
    'low': 30.0,          # 30조원+
    'very_low': 0.0,      # 30조원 미만: 자금 이탈 (위험)
}

# 예탁금 변화율 시그널
DEPOSIT_CHANGE_SIGNALS = {
    'surge': 0.10,        # MoM +10%+: 대기 자금 급증 (매수 대기)
    'increase': 0.03,     # MoM +3%~+10%
    'stable': -0.03,      # MoM -3%~+3%
    'decrease': -0.10,    # MoM -3%~-10%
    'exodus': -0.10,      # MoM < -10%: 자금 이탈 (위험)
}
```

#### 새 함수

```python
def analyze_investor_deposits(deposit_balance_t, deposit_change_1m_pct=0.0):
    """고객 예탁금(투자자예수금) 독립 분석.

    Args:
        deposit_balance_t: float, 현재 예탁금 (조원).
        deposit_change_1m_pct: float, 1개월 변화율 (%, e.g. 5.0 = +5%).

    Returns:
        dict: {
            balance_t: float,
            level: str,  # very_high/high/normal/low/very_low
            change_pct: float,
            signal: str,  # surge/increase/stable/decrease/exodus
            interpretation: str,  # 한글 해석
            buying_power_score: float,  # 0-100 매수 여력 점수
        }
    """
```

| deposit_balance_t | level | 의미 |
|:--:|:--:|------|
| >= 60조 | very_high | 대기 자금 풍부 |
| >= 50조 | high | 양호 |
| >= 40조 | normal | 보통 |
| >= 30조 | low | 부족 |
| < 30조 | very_low | 자금 이탈 |

| deposit_change_1m_pct | signal | 의미 |
|:--:|:--:|------|
| >= +10% | surge | 대기 자금 급증 |
| >= +3% | increase | 증가 |
| >= -3% | stable | 안정 |
| >= -10% | decrease | 감소 |
| < -10% | exodus | 자금 이탈 |

**buying_power_score 계산**:
- level_score: very_high=90, high=70, normal=50, low=30, very_low=10
- signal_score: surge=90, increase=70, stable=50, decrease=30, exodus=10
- `buying_power_score = level_score * 0.5 + signal_score * 0.5`

### 3.3 `forced_liquidation_risk.py` 연동 (선택적)

기존 `CREDIT_RISK_WEIGHTS` 4-컴포넌트는 변경하지 않음. 예탁금 분석은 **독립 함수**로 제공하여, 리포트에서 별도 섹션으로 출력.

> 이유: 기존 가중치(0.30+0.25+0.25+0.20=1.00)를 변경하면 기존 점수가 달라짐. 예탁금은 "리스크"가 아닌 "매수 여력" 지표이므로 별도가 적절.

### 3.4 테스트 설계 (~10개)

| 테스트 | 검증 내용 |
|--------|----------|
| `test_deposit_level_very_high` | 65조 → very_high |
| `test_deposit_level_normal` | 42조 → normal |
| `test_deposit_level_very_low` | 25조 → very_low |
| `test_deposit_signal_surge` | +12% → surge |
| `test_deposit_signal_stable` | +1% → stable |
| `test_deposit_signal_exodus` | -15% → exodus |
| `test_buying_power_score_range` | 0 <= score <= 100 |
| `test_buying_power_high_balance_surge` | 65조+12% → score >= 80 |
| `test_buying_power_low_balance_exodus` | 25조-15% → score <= 20 |
| `test_deposit_interpretation_korean` | interpretation이 한글 문자열 |

## 4. Item 3: PANIC_BUY 패턴 — `pattern_classifier.py`

### 4.1 현재 구조

```python
MARKET_PATTERNS = {
    'POLICY_PIVOT': {..., 'equity_range': (70, 90)},
    'UNSUSTAINABLE_DISTORTION': {..., 'equity_range': (30, 50)},
    'EXTREME_CONTRARIAN': {..., 'equity_range': (25, 40)},
    'WAIT_OBSERVE': {..., 'equity_range': (0, 20)},
}
# 우선순위: POLICY_PIVOT > UNSUSTAINABLE > EXTREME_CONTRARIAN > WAIT
```

### 4.2 변경 설계

#### 새 패턴 추가

```python
MARKET_PATTERNS['PANIC_BUY'] = {
    'name': '공포 매수 기회',
    'trigger': 'easing_regime + sharp_decline + no_forced_liquidation',
    'principle': '금리 인하기의 급락은 최고의 매수 기회',
    'equity_range': (80, 100),
}
```

#### 트리거 조건 (3개 동시 충족)

```python
PANIC_BUY_TRIGGERS = {
    'us_regime_min_score': 65,         # US regime score >= 65 (easing)
    'market_decline_pct': -10.0,       # KOSPI 20일 수익률 <= -10%
    'vkospi_min': 25.0,                # VKOSPI >= 25 (공포 확인)
}
```

| 조건 | 파라미터 | 기준값 | 소스 |
|------|---------|--------|------|
| US 완화 환경 | us_regime_score | >= 65 | `us-monetary-regime` report |
| 시장 급락 | kospi_20d_return | <= -10% | `kr-market-breadth` 또는 직접 |
| 공포 확인 | vkospi | >= 25 | `kr-bubble-detector` 또는 직접 |

#### 우선순위 변경

```
PANIC_BUY > POLICY_PIVOT > UNSUSTAINABLE > EXTREME_CONTRARIAN > WAIT
```

PANIC_BUY가 최고 우선순위. 이유: 금리 인하기의 급락은 드물지만 가장 강한 시그널.

#### 새 함수

```python
def _check_panic_buy(components, reports):
    """공포 매수 기회 패턴 감지.

    조건 (3개 동시):
    1. US regime score >= 65 (easing)
    2. KOSPI 20일 수익률 <= -10%
    3. VKOSPI >= 25

    Returns:
        tuple: (detected: bool, confidence: float)
    """
```

confidence 계산:
- base = 70
- US regime이 80+ (강한 easing)이면 +10
- 하락폭 -15% 이하이면 +10
- VKOSPI 30+ (극심한 공포)이면 +10
- max = 100

#### `classify_pattern()` 변경

```python
def classify_pattern(components, reports):
    # 우선순위 1: PANIC_BUY (NEW)
    detected, confidence = _check_panic_buy(components, reports)
    if detected:
        p = MARKET_PATTERNS['PANIC_BUY']
        return {...}

    # 우선순위 2: POLICY_PIVOT (기존)
    ...
```

### 4.3 `conviction_scorer.py` 연동

PANIC_BUY 감지 시 확신도 보너스:

```python
# conviction_scorer.py의 calc_conviction_score() 또는 외부 호출에서
# pattern == 'PANIC_BUY'이면 score += 15 (cap at 100)
```

> 구현 위치: `conviction_scorer.py`의 `calc_conviction_score()`에 optional `pattern` 파라미터 추가하거나, 상위 오케스트레이션에서 처리.

**설계 결정**: `calc_conviction_score(components, pattern=None)` — pattern이 PANIC_BUY이면 최종 점수에 +15 보너스 (최대 100).

### 4.4 테스트 설계 (~10개)

| 테스트 | 검증 내용 |
|--------|----------|
| `test_panic_buy_pattern_in_dict` | MARKET_PATTERNS에 PANIC_BUY 존재 |

| `test_panic_buy_all_conditions_met` | 3조건 충족 → detected=True |

| `test_panic_buy_no_decline` | decline > -10% → detected=False |

| `test_panic_buy_highest_priority` | PANIC_BUY가 POLICY_PIVOT보다 우선 |
| `test_panic_buy_confidence_base` | 기본 3조건 충족 → confidence >= 70 |
